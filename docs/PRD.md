# 專案名稱：食譜收藏系統

## 1. 專案概述

### 背景與動機
現代人常常有許多散落各處的食譜記錄（像是社群媒體的貼文、截圖或是手寫筆記），在需要烹飪時往往難以快速找到想要的食譜，甚至會有食材不知如何應用的困擾。因此，開發「食譜收藏系統」旨在提供一個統一的數位化平台，讓使用者能井然有序地整理自己的私房菜單與收集來的料理靈感。

### 目標用戶
- 主要：個人用戶（我自己），以及喜歡在家開伙、紀錄食譜的料理愛好者。

### 核心價值主張
一個清晰、易查、好用的個人化食譜庫，不僅能有條理地紀錄與搜尋食譜，更能聰明地依據手邊現有食材給予推薦，並自動產生採買清單，大幅降低下廚前的準備負擔。

---

## 2. 功能需求

### 2.1 主要功能清單
1. 新增食譜
2. 搜尋食譜
3. 根據食材推薦食譜
4. 標籤與分類管理
5. 採購清單自動生成

### 2.2 使用者故事 (User Stories)

| 功能模組 | 使用者故事 |
| --- | --- |
| **新增食譜** | 作為 **使用者**，我希望 **能輸入食譜的名稱、所需食材、步驟和圖片並儲存**，以便 **記錄我新學到的料理**。 |
| **搜尋食譜** | 作為 **使用者**，我希望 **能透過關鍵字（如菜名）搜尋系統內的食譜**，以便 **快速找出今晚想煮的那道菜**。 |
| **根據食材推薦** | 作為 **使用者**，我希望 **能輸入目前冰箱裡剩餘的食材**，以便 **系統推薦適合烹調的食譜，解決剩菜煩惱**。 |
| **標籤與分類管理** | 作為 **使用者**，我希望 **能為食譜加上自訂標籤（如：台式、甜點、快速料理）**，以便 **將食譜分門別類，未來瀏覽時更有條理**。 |
| **自動生成採購清單** | 作為 **使用者**，我希望 **在選定要煮的食譜後，能自動將缺少的食材整理成清單**，以便 **我去超市採購時能直接看清單買齊**。 |

---

## 3. 非功能需求

### 3.1 技術限制
- **Backend framework**: Flask
- **Template engine**: Jinja2
- **Database**: SQLite
- **Architecture**: 輕量級單體應用架構 (Monolithic Architecture)

### 3.2 效能與安全考量
- **效能考量**：
  - 由於以個人或小群體使用為主，SQLite 的讀寫效能已能滿足需求。
  - 食譜圖片上傳應限制檔案大小，並進行適當的壓縮或轉換，減少磁碟空間佔用與讀取時間。
- **安全考量**：
  - 防止 SQL Injection：使用適當的 ORM (如 SQLAlchemy) 或參數化查詢。
  - XSS 防護：在 Jinja2 中確保內容有被正確 escape，特別是使用者輸入的食譜步驟或描述。

## 4. MVP 範圍 (Minimum Viable Product)

| 優先級 | 功能敘述 | 說明 |
| --- | --- | --- |
| **Must Have (核心必備)** | - 新增、編輯、刪除食譜<br>- 瀏覽所有食譜列表<br>- 基礎關鍵字搜尋 | 這是食譜收藏最基礎的需求，滿足這些功能系統才有價值。 |
| **Should Have (重要擴展)** | - 標籤與分類管理功能<br>- 根據食材反向搜尋/推薦食譜 | 分類可以大幅提昇整理的體驗；食材反查能解決「不知道該煮什麼」的痛點。 |
| **Nice to Have (錦上添花)** | - 圖片上傳功能<br>- 自動生成採購清單<br>- 份量換算功能 | 對個人下廚更有幫助的輔助功能，但於初期開發時可暫緩實作。 |
